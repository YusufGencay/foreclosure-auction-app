# Florida Foreclosure Auction Analysis & Ranking Tool — Project Context

## What this is
A decision-support web app for a real estate investor (Yusuf) who bids at Florida foreclosure auctions. It aggregates upcoming foreclosure auction listings from FL county clerk sites, scores/ranks properties against investment criteria, and provides a single dashboard to triage properties quickly.

## Geographic scope (config-driven, in `config/counties.yaml`)
- Tampa Bay: Hillsborough, Pinellas, Pasco, Hernando, Manatee, Sarasota
- Central FL: Orange, Osceola, Seminole, Polk, Lake, Volusia, Brevard, Marion
- CORRECTED 2026-07-05 (per user, confirmed live via browser): 12/14 counties run active mortgage-foreclosure calendars on RealAuction (realforeclose.com). Seminole IS one of them (seminole.realforeclose.com - live, real scheduled sales) - an earlier version of this doc wrongly claimed Seminole used a custom webapps.seminoleclerk.org system; that was never actually verified live and has been corrected. Lake and Osceola are the only 2 exceptions: their realforeclose.com foreclosure calendars are confirmed offline ("This feature is currently offline"); both instead run active tax deed sales on realtaxdeed.com, which is what config/counties.yaml now points at for those two.

## Tech stack
Python FastAPI backend + React/Vite frontend + SQLite DB. APScheduler for daily scrape scheduling. Single-service Docker deploy (FastAPI serves the built React app as static files under one process, API under `/api/*`).

## Where the code lives
Local folder: `foreclosure-app/` inside the Cowork outputs folder (persists on Yusuf's Mac). Contains `backend/`, `frontend/`, `config/`, `Dockerfile`, `railway.toml`, `.env.example`, `README.md`.

## Comprehensive live bug sweep (2026-07-05, evening) — root causes found and fixed
User reported broad data-quality complaints ("almost everything is not working"). Did a systematic live-DOM/live-network investigation (real Chrome browser, not this sandbox) rather than only checking the specific items raised, and found and fixed five distinct real bugs, all pushed live and confirmed deployed:

1. **`realauction_playwright.py` was silently dropping most of a day's auctions and mixing in closed/canceled ones.** Live DOM inspection of hillsborough.realforeclose.com showed `.AUCTION_ITEM` matches items in both `#Area_W` (real upcoming auctions) and `#Area_C` (already-closed/canceled) with no distinction, and the "Auctions Waiting" list paginates via an AJAX-in-place `.PageRight` control that the scraper never clicked — confirmed on a real 15-active-auction day that only 10 were being captured (page 1 only) and closed sales could leak in as if upcoming. Fixed: scoped extraction to `#Area_W`, tag+drop `#Area_C` items, added a bounded pagination loop keyed on `#curPWA`/`#maxWA` (capped at `MAX_PAGES_PER_DAY=20`), falls back to old single-page/unscoped behavior if that DOM structure isn't present.
2. **Zillow/Realtor.com/Redfin scrapers were guessing URLs that don't work.** Live-checked: `zillow.com/homes/<address>_rb/` and `zillow.com/homedetails/<address>/` (no zpid) both silently redirect to a generic rental search with zero relation to the requested address; Redfin's `/search?query=...` redirects to the homepage. Fixed: Zillow/Realtor.com now resolve the address to its real canonical detail-page URL (with the required internal zpid/listing-id) via a `site:<domain> <address>` query against `html.duckduckgo.com`'s no-JS HTML search endpoint, which reliably surfaces the correct URL as its top hit (confirmed live for a real address on both sites). Redfin uses its own `stingray/do/location-autocomplete` API first (also confirmed live) with the same DuckDuckGo approach as a fallback.
3. **Estimate scrapers were hanging on `wait_until="networkidle"`.** Confirmed live in production logs: both Zillow and Redfin never go fully network-idle (continuous background polling/analytics), so every real `/enrich` call timed out at 30s before the page even rendered — no CAPTCHA ever detected because the page never loaded. Fixed: switched to `wait_until="domcontentloaded"` + a short explicit settle wait, the standard fix for this class of issue.
4. **The dollar-figure extraction window was too tight.** Live-checked Realtor.com's real rendered page: the "RealEstimate" label and its dollar figure are ~90 characters apart once flattened to plain text (interleaved chart/table UI text), but the regex only tolerated 40 characters, so a real, present estimate would have been missed. Widened to 400 characters.
5. **The raw-JSON fetch path (Redfin's autocomplete API, the DuckDuckGo search) had a render race.** Confirmed live: `page.goto(url)` + `page.inner_text("body")` is meant for rendered HTML pages, but for a raw JSON API response Chromium renders its own JSON-viewer UI over it, and reading that back raced with the viewer in production (`json.JSONDecodeError: Extra data...` on a response that was the correct literal text when checked interactively). Fixed: switched to Playwright's dedicated API request context (`playwright.request`), which does a plain HTTP GET with no DOM/rendering step at all - faster and immune to this race.
6. **`POST /api/scrape/all` was 404ing in production the entire time** (found while investigating why 13/14 counties had never scraped). Root cause: `POST /api/scrape/{county}` was registered in `main.py` *before* `POST /api/scrape/all`, so FastAPI/Starlette (which matches path routes in registration order) matched `{county}="all"` first, looked up a county literally named "all", found none, and raised a 404 — logged as a bare "404 Not Found" with no indication this was a routing bug rather than an intentional not-found. This is the actual reason the other 13 counties never got their first scrape: the only thing that ever successfully ran `scrape_all_counties()` was the scheduled 06:00/18:00 APScheduler job, which calls that Python function directly rather than going through this HTTP route. Fixed by moving the exact-literal route above the parameterized one.
